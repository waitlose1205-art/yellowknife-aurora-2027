# 極光旅行推薦系統：模組化後續工作分流

更新日：2026-07-20

本文件是內部開發順序，不是公開網頁內容。目的在於避免把每日更新、多旅行社解析、航班住宿補全、自由行估算一次塞進同一個腳本或同一個頁面。

## 模組拆分原則

1. 公開網頁只讀整理後的靜態資料，不讓訪客觸發旅行社查詢。
2. 後台資料更新、來源解析、航班住宿補全、自由行估算必須分成不同工作流。
3. 每個工作流都要有輸入、輸出、查核規則與失敗降級方式。
4. 未通過來源驗證或欄位查核的資料不可進入推薦排序，只能保留在內部查核輸出。

## 第一分流：每日更新

目的：讓資料每天自動重新產生，避免旅行團日期、價格、報名狀態與來源頁過期。

目前狀態：
- 已有 `npm run data:daily`，可執行含商品頁補查的資料產生流程。
- 已有 `scripts/audit-tour-data-readiness.mjs`，可輸出資料品質查核。
- 尚未接成正式每日排程。

啟用條件：
- 資料產生腳本完成模組拆分，失敗時不會中斷整批資料。
- 查核腳本能阻擋高風險資料進入公開 JSON。
- 每次更新可輸出摘要：商品總數、來源數、高風險數、航班完整數。

建議輸出：
- `public/data/tour-products.latest.json`
- `public/data/source-status.json`
- `exports/tour-data-readiness-audit-YYYY-MM-DD.json`
- `exports/tour-data-readiness-audit-YYYY-MM-DD.md`

下一步：
- 建立 `scripts/pipeline/`，把讀取、補查、查核、輸出拆成可重用步驟。

## 第二分流：多旅行社解析

目的：讓每家旅行社有自己的解析器，避免所有站台規則混在同一支大型腳本。

目前狀態：
- `scripts/build-tour-data.mjs` 已含雄獅、可樂、鳳凰、喜鴻等站台的特殊處理。
- 站台差異很大，有些可直接讀 HTML，有些需要瀏覽器渲染或隱藏 JSON。

啟用條件：
- 每家旅行社至少有一個 adapter：輸入來源列或來源網址，輸出標準商品欄位。
- adapter 失敗時必須回傳原因，不可捏造航班、價格或商品名稱。
- 來源頁標題與商品名稱比對需保留分數與狀態。

建議模組：
- `scripts/pipeline/agencies/liontravel.mjs`
- `scripts/pipeline/agencies/colatour.mjs`
- `scripts/pipeline/agencies/travel4u.mjs`
- `scripts/pipeline/agencies/besttour.mjs`
- `scripts/pipeline/agencies/phoenix.mjs`
- `scripts/pipeline/agencies/generic-html.mjs`

下一步：
- 先抽出雄獅與喜鴻，因為它們最能代表「隱藏 JSON」與「瀏覽器渲染補資料」兩種高風險模式。

## 第三分流：航班住宿補全

目的：把目前的航班摘要提升成可比較資料，並補足飯店、住宿月份價格與缺資料狀態。

目前狀態：
- `flightSummary` 已進入公開資料。
- 航班欄位仍是文字摘要，部分商品只有航空公司代碼或官方未揭露說明。
- 住宿資料尚未形成獨立欄位。

啟用條件：
- 航班資料需拆成結構化欄位：航空、去程、回程、轉機點、起降時間、是否完整。
- 住宿資料需拆成飯店名稱、城市、月份價格、是否官方揭露、是否推估。
- 商品頁未公告時，必須顯示「尚未公布」，不可用自由行估算替代團體商品資料。

建議資料欄位：
- `flightSegments`
- `flightCompleteness`
- `hotelOptions`
- `hotelRateMonth`
- `hotelDisclosureStatus`

下一步：
- 先建立欄位 schema 與查核規則，再逐步讓旅行社 adapter 補欄位。

## 第四分流：自由行估算

目的：在旅行團商品之外，建立自由行的參考成本區間，包含機票、住宿、當地極光活動與保險雜費。

目前狀態：
- 自由行概念已在需求中確認，但目前公開資料仍以旅行團商品為主。
- 自由行不能只用團費推估，必須有航班、住宿、月份價格與風險緩衝。

啟用條件：
- 航班與飯店資料已能獨立輸入，不依賴旅行團商品欄位。
- 匯率、稅費、行李、冬衣、保險、當地活動可分項顯示。
- 能區分「官方查價」、「第三方參考價」、「歷史樣本估算」。

建議輸出：
- `public/data/independent-travel-estimates.latest.json`
- 每個月份一組：機票低中高、飯店低中高、活動、保險雜費、總額區間。

下一步：
- 先用 2026 旅行團航班與住宿資訊建立參考模板，再加入官方航班/飯店查價資料源。

## 推進順序

1. 先穩定每日更新資料流，避免公開資料過期。
2. 再拆旅行社 adapter，降低單一來源修正造成全站資料壞掉的風險。
3. 接著補航班住宿結構化欄位，讓商品比較更完整。
4. 最後啟用自由行估算，避免在基礎資料不足時產生誤導性金額。
